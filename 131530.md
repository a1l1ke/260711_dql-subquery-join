### 📋 개념 비교 표
| 단계 | 일반 SQL 실행 순서의 모순 | 인라인 뷰(FROM 절 서브쿼리)를 통한 우회 |
| :--- | :--- | :--- |
| **1단계** | `FROM` 절에서 원본 데이터 로드 | `FROM` 절의 서브쿼리를 먼저 돌려 **가공된 컬럼(`PRICE_GROUP`)을 포함한 임시 테이블**을 메모리에 올림 |
| **2단계** | `GROUP BY` 절 수행 (아직 `SELECT` 연산이 안 되어 별칭 인지 불가) | 메인 쿼리의 `GROUP BY` 절 수행 시, 이미 하위 서브쿼리에서 계산이 끝난 `PRICE_GROUP` 별칭을 안전하게 인지함 |
| **3단계** | `SELECT` 절 수행 (여기서 드디어 가공 및 별칭 선언) | 이미 분류와 집계(`COUNT(*)`)가 끝난 최종 결과 데이터를 화면에 매핑함 |

## 2. ⚙️ 주니어를 위한 원리 및 구조 설명

### 🔄 SQL 실행 순서와 SELECT Alias의 한계

표준 SQL의 논리적 쿼리 처리 순서(Logical Query Processing Order)는 다음과 같습니다.

```
FROM ➡️ WHERE ➡️ GROUP BY ➡️ HAVING ➡️ SELECT ➡️ ORDER BY
```

#### 🚨 에러 발생 원인 (표준 SQL 규격 기준)
```sql
-- [표준 SQL에서 에러가 나는 형태]
SELECT FLOOR(PRICE / 10000) * 10000 AS PRICE_GROUP, COUNT(*)
FROM PRODUCT
GROUP BY PRICE_GROUP; -- ❌ 표준 SQL 에러! (PRICE_GROUP을 인식할 수 없음)
```
* **이유**: `GROUP BY` 단계는 3번째 순서인 반면, `PRICE_GROUP`이라는 별칭이 정의되는 `SELECT` 단계는 5번째 순서입니다. 엔진은 아직 생성되지도 않은 컬럼명(`PRICE_GROUP`)으로 그룹화를 시도하기 때문에 문법 에러가 발생합니다.

#### 💡 인라인 뷰를 통한 해결 원리
[131530.sql:L18-27](file:///Users/morgan/Documents/workspace/260711_dql-subquery-join/131530.sql#L18-27)에 제시된 정석적인 해결 기법입니다.
```sql
SELECT PRICE_GROUP, COUNT(*) AS PRODUCTS
FROM (
    -- 안쪽 서브쿼리(인라인 뷰)가 FROM 절에서 먼저 연산되어 파생 테이블을 만듦
    SELECT FLOOR(PRICE / 10000) * 10000 AS PRICE_GROUP
    FROM PRODUCT
) P -- 파생 테이블 별칭 P 지정
GROUP BY PRICE_GROUP; -- 바깥 메인 쿼리에서는 이미 확정된 PRICE_GROUP을 참조함
```
* **원리**: `FROM` 절 내부에 서브쿼리를 작성하면, 이 서브쿼리가 메인 쿼리의 `GROUP BY`보다 먼저 완전히 가공 및 계산을 마칩니다. 따라서 바깥 메인 쿼리가 실행되는 시점에는 이미 `PRICE_GROUP`이라는 열이 실존하는 컬럼처럼 임시 테이블에 존재하므로 완벽히 동작합니다.

### 🧮 수치 절사 함수 비교

가격을 만 원 단위로 구간화하기 위해 소수점 버림 및 절사 함수가 사용됩니다.

| 함수명 | 작동 특징 | 예시 입력 | 예시 출력 |
| :--- | :--- | :--- | :--- |
| **`FLOOR(val)`** | 주어진 숫자보다 **크지 않은 가장 큰 정수** 반환 (소수점 이하 내림) | `FLOOR(1.8)` <br> `FLOOR(-1.2)` | `1` <br> `-2` (음수 방향 하향) |
| **`TRUNCATE(val, len)`** | 지정된 자릿수 이하를 **단순 삭제(버림)** | `TRUNCATE(15400, -4)` <br> `TRUNCATE(1.8, 0)` | `10000` (백의 자리 이하 버림) <br> `1` |
| **`ROUND(val, len)`** | 지정된 자릿수에서 **반올림** 수행 | `ROUND(15400, -4)` | `20000` |

> [!TIP]
> **가격 구간 계산 시 선택**
> 음수 금액이 없는 상품 가격의 경우 `FLOOR(PRICE / 10000) * 10000` 공식과 `TRUNCATE(PRICE, -4)` 공식 모두 동일하게 만 원 단위 절사 기능을 수행합니다. 단, 수학적 정합성과 음수 처리 안정성을 고려한다면 `TRUNCATE`가 직관적이고 표준에 가깝습니다.

## 3. 🎯 SQLD 자격증 대비 핵심 이론

### ⚠️ 파생 테이블(Derived Table)의 별칭 필수성
SQLD 시험 및 실무 시험의 감점 포인트입니다.
`FROM` 절에 서브쿼리를 사용하는 인라인 뷰의 경우, **반드시 임시 테이블의 별칭(Alias)을 작성해야만 합니다.**
* **에러 코드**: `SELECT * FROM (SELECT col FROM tab);` ➡️ `Every derived table must have its own alias` 에러 발생.
* **정상 코드**: `SELECT * FROM (SELECT col FROM tab) AS temp;` (테이블 끝에 별칭 `temp`를 기입하여 객체화)

## 4. 📝 면접 대비 예상 질문 & 답변 (Q&A)

### Q1. SELECT 절에서 정의한 별칭(Alias)을 GROUP BY 절에서 참조할 수 없는 이유와, MySQL에서 이것이 예외적으로 동작하는 이유에 대해 설명해 주세요.
**A1.**
* SQL의 논리적 실행 순서상 `GROUP BY`가 `SELECT` 절보다 먼저 실행되기 때문에 표준 SQL에서는 `SELECT` 절의 Alias를 `GROUP BY`에서 사용할 수 없습니다.
* 하지만 MySQL은 자체 옵티마이저가 쿼리 파싱 단계에서 `SELECT` 절의 Alias를 인지할 수 있도록 컴파일 과정을 내부적으로 확장 지원하여 예외적으로 동작을 허용합니다. 다만 이는 표준 SQL 규격이 아니므로, 타 DBMS(Oracle 등)로 이식할 경우 심각한 문법 오류를 유발하므로 인라인 뷰를 사용하는 편이 안전합니다.

---

### Q2. 인라인 뷰(Inline View)를 과도하게 사용했을 때 발생할 수 있는 문제점은 무엇인가요?
**A2.**
인라인 뷰는 논리적으로 가상의 테이블을 만드는 행위입니다.
서브쿼리가 복잡하거나 데이터량이 많을 경우, DBMS가 이 인라인 뷰를 메모리나 디스크에 임시 테이블로 구체화(Materialization)하게 되며, 이로 인해 인덱스를 타지 못하고 **임시 테이블 생성 비용 및 추가 풀 스캔 부하가 발생**하여 성능이 악화될 수 있습니다. 현대 옵티마이저가 인라인 뷰를 메인 쿼리와 병합(View Merging)하여 조인문으로 최적화해주기도 하지만, 복잡도가 높다면 `WITH` 절(CTE)로 정리하거나 조인 구조로 재작성해야 합니다.

## 5. 🛠️ 일반화 및 추상화된 인라인 뷰 가공 템플릿

특정 수치를 구간 단위로 그룹화하여 카운트하는 정석적인 템플릿입니다.

```sql
-- 가공 컬럼 기반 그룹화 및 집계 템플릿
SELECT
    temp.bucket_key AS range_group,
    COUNT(*)        AS total_count
FROM (
    -- [인라인 뷰]: 특정 조건으로 구간화(Bucket) 가공한 뒤 별칭 부여
    SELECT
        TRUNCATE([NUMERIC_COLUMN], [INTERVAL_DECIMAL_PLACE]) AS bucket_key
    FROM
        [TARGET_TABLE]
) AS temp -- 파생 테이블 고유 별칭 필수
GROUP BY
    temp.bucket_key -- 가공된 별칭으로 안전하게 분류
ORDER BY
    range_group ASC; -- 가격대별 정렬
```
